- [2026-07-07] Don't mock a module-class constructor with `vi.fn().mockImplementation(() => obj)` when the module code does `new SomeCtor(...)` (bug-140). `vi.mock("@x/y", () => ({ Ctor: mocks.Ctor.mockImplementation(...) }))` produces `TypeError: () => ({...}) is not a constructor` at the `new` site because vitest's mock function isn't constructable. The pattern from `llm-client.test.ts` works: declare the mock factory as a real ES `class` with the methods/fields the consumer touches, capture the constructor opts, and `vi.fn()` only the inner behaviour (e.g. `generateContent`). To assert that `new Ctor(opts)` was called, save the opts inside the class `constructor` to a closure-backed `vi.hoisted` ref.

- [2026-07-07] Don't `instanceof ApiError` to detect errors thrown by `@google/genai` in tests/mocks (bug-141). The SDK uses a single `ApiError` class for every HTTP failure (auth, rate-limit, model-not-found, upstream), distinguished only by `.status`. In test mocks the SDK isn't really installed, so `instanceof` returns false even on shapes that would be `ApiError`s at runtime. Detect by shape instead: `const looksLikeSdkApiError = (e): e is { status: number } & Error => typeof e?.status === "number"`. Same applies to any SDK with a typed error class that may be loaded dynamically or swapped in tests.

- [2026-07-07] Don't pass real-looking URLs (`https://x/y.jpg`) to a client helper in unit tests without stubbing global `fetch` (bug-139). The helper will hit the real network and fail with `TypeError: fetch failed ... ENOTFOUND`. Either pass a `data:` URI (so the helper short-circuits its own fetch), or wrap the test block with `vi.stubGlobal("fetch", vi.fn().mockResolvedValue({ ok: true, arrayBuffer: () => Promise.resolve(new Uint8Array([1,2,3])) }))` and `vi.unstubAllGlobals()` in `afterEach`. This applies to any test that exercises a code path which `await fetch(url)` — `vi.fn()` on the imported client doesn't intercept the global.
- [2026-07-04] **Domain errors from `@clipflow/youtube-upload` MUST be mapped to `AppError` at the service boundary.** `PermanentPublishError` / `TransientPublishError` are not `AppError`s, so the central error middleware in `apps/api/src/middleware/error.ts` treats them as "unknown" → generic 500 "Something went wrong". The real YouTube reason (quota, needs-reauth, invalid metadata, channel-not-connected, etc.) is logged server-side but never crosses the wire. Fix in `videos.service.ts`: `mapYouTubeErrorToAppError` + `PERMANENT_REASON_TO_APP` table wraps every YouTube call in the service layer. Any NEW service that calls the youtube-upload package (or any future domain package that throws its own error classes) must do the same — either extend the mapper or write a per-module equivalent. Don't teach the central middleware about domain-specific error types.

- [2026-06-28] Cache abstraction uses two backends (Redis when REDIS_URL is set, in-memory fallback). The exported `cache` singleton is a delegating object — `get/set/del` forward to whichever backend `initCache(env)` selected — so every service call site (`cache.get("settings:${userId}")` in settings/youtube/videos) keeps working unchanged. Optional services (in-memory cache, queue when REDIS_URL unset) show as "skipped". Worker side opens a fresh `ioredis` probe connection (separate from the BullMQ one) so the PING is independent of queue lifecycle.

- [2026-06-28] ioredis needs `lazyConnect: true` when you want to control connection timing at boot. Combined with `maxRetriesPerRequest: null` (required for BullMQ compatibility), this lets you call `redis.connect()` + `redis.ping()` once at startup, fail fast if Redis is unreachable, and never accidentally issue a command before the socket is up. The `error` event handler must be a no-op (or log) — ioredis will emit errors mid-reconnect and you don't want them to crash the process.

- [2026-06-30] **pnpm + Docker pitfall.** pnpm uses isolated mode by default — each workspace package's direct deps live under `apps/<pkg>/node_modules/<dep>` as *relative symlinks* into the virtual store at `node_modules/.pnpm/`. A Dockerfile that copies only the root `/app/node_modules` (without `apps/<pkg>/node_modules`) will let the runtime process find `.pnpm/<dep>` but NOT `<dep>` itself — Node walks up looking for `node_modules/<dep>` from the entrypoint and finds nothing. Symptom: `ERR_MODULE_NOT_FOUND: Cannot find package '<dep>'`. Fix: every Dockerfile runtime stage that runs a pnpm-managed workspace package must `COPY --from=build apps/<pkg>/node_modules ./apps/<pkg>/node_modules` alongside the root `node_modules`. Either that, or `pnpm deploy` (single self-contained output dir), or `.npmrc` `public-hoist-pattern[]=<dep>` to hoist specific runtime deps to the root. The bug is latent until the package gains a direct import of one of its deps — BullMQ was previously imported indirectly (BullMQ's own nested node_modules carried `ioredis`), so it shipped fine without the apps/worker/node_modules layer; once apps/worker/src/index.ts started doing `import { Redis } from "ioredis"` directly, the missing COPY became fatal.

- [2026-07-01] **vitest `vi.mock` factory TDZ bug** (bug-071). `vi.mock("@module", () => ({ fn: mockFn }))` is hoisted above module-level `const mockFn = vi.fn()` declarations by vitest's transformer. - [2026-07-01] **Worker-side enqueue vs API-side enqueue for follow-up jobs**. The v1.5 ingest → transcription pipeline enqueues `transcribe-${videoId}` from inside the `video-ingest` BullMQ worker (after the row is flipped to `TRANSCRIBING`), NOT from the API at video-create time. Reason: with concurrency=1 on each queue, a worker's own follow-up fires only AFTER the parent completes, so the deterministic `s3KeyAudio` guarantee holds without any polling. If the API chained them, the transcription job would either (a) need to poll for `s3KeyAudio` (ugly), or (b) race against the still-running ingest and fail permanent with `[AAI_AUDIO_MISSING]` (worse — it would cause false-positive permanent failures). The worker-side `enqueueTranscription` callback is passed through `ProcessContext` and is optional so existing tests can pass a no-op. API-side `enqueueTranscriptionJob` still exists in `apps/api/src/lib/queue.ts` for admin-style retry paths (e.g. retry-from-FAILED).

- [2026-07-01] **Dashboard needs an SSE fallback, not just SSE.** The dashboard's `useVideoSSE` is the real-time path, but it's a single point of failure: if Redis blips, EventSource silently disconnects without an obvious UI signal, or the worker/API channel-prefix drifts, the row stays stuck on its old badge forever (bug-081). Solution: pair SSE with a `setInterval` safety-net refetch, gated on `hasInFlightVideo` (any non-final status). 15 s feels live without hammering the server. Interval clears the moment no in-flight rows remain — free in the steady state. Always `useMemo` the array dependency so the polling effect doesn't tear down + recreate every render.

- [2026-07-01] **Single LLM call for {summary, chapters[]} beats two separate calls.** Putting summary + chapter selection in one prompt cuts latency in half, costs roughly the same as one larger prompt (vs. two), and keeps the chapter boundaries + summary internally consistent — they reason over the same context in the same forward pass. The output schema `LlmOutputSchema` enforces YouTube rules (first chapter = 0, ≥10 s gap, ≤12 chapters, ≤100 char titles, ≤280 char summary) server-side via zod `.refine()` so the LLM gets a focused system prompt and we get the safety net.

- [2026-07-01] **LLM retries belong in the caller, not the client.** The OpenAI SDK wrapper (`OpenAICompatLlmClient.complete`) is a thin facade: it makes one call, unwraps the text, surfaces the SDK error class. It does NOT classify or retry. Classification (`classifyLlmError` → `{ kind, reasonCode, message }`) and parse-retry (`validateWithRetry`, max 3 attempts with the rejection reason appended to the next userPrompt) both live in the `generate` job's call site. The thin client is trivially mockable; the caller owns the "permanent vs transient" decision.

- [2026-07-01] **`chaptersJson` schema changed shape mid-flight** (no migration needed). Originally declared as a bare chapter array. v1.5 adds a `summary` field alongside chapters — so it becomes `{ summary, chapters[] }`. The publish path originally cast it as the bare array directly (WRONG — see bug-125); the LLM path writes both fields (one `LlmOutputSchema.safeParse` succeeds, single JSON column write); the API's `updateVideo` rebuilds the wrapper from `existing.chaptersJson ?? { summary: "", chapters: [] }` so partial saves don't clobber. Why no migration: Prisma's `Json?` column accepts any JSON shape; only the schema comment + TypeScript type changed. If we ever want to validate the JSON shape at the DB layer, switch to Postgres's typed JSON or zod on read at every consumption point.

- [2026-07-04] **`chaptersJson.chapters` is the canonical chapter list, not `chaptersJson` itself** (bug-125). Always pass the inner array to `formatChaptersForDescription` — passing the wrapper object lands you in the trap where `chapters.length` is `undefined`, the `< 3` guard fails open, and you map over the object's string + array keys producing `NaN:NaN` lines that YouTube silently ignores. Today the helper is shape-tolerant (accepts both the bare array and the wrapper) AND the publish site extracts the inner array via `extractChapters()` — two layers of defense. The Web side (review components) already reads `chaptersJson.chapters` correctly — only the publish path was broken. Whenever you add a new consumer of `chaptersJson`, prefer reading `.chapters` explicitly, not the wrapper.

- [2026-07-01] **vitest `mock.calls` destructure off-by-one** (bug-085). `mock.calls[0]` for a single-arg call is `[{...}]` — one inner array containing one element. `[, args]` skips that element and captures index 1 (always `undefined`). The bug hides in `.filter()` calls (predicate is false for `undefined.args.data`, filter returns `[]`, `toHaveLength(0)` passes by luck). It surfaces in `.map()` calls. Single-arg mock calls should always use `([args])`, not `([, args])`. Audit pattern when reviewing test files.

- [2026-07-02] **Partial-update endpoints must check `'key' in input`**, not `input.key !== undefined`, so explicit `null` survives the merge (a creator clearing the description to `null` shouldn't accidentally leave the old text in place). Mirror the `useUpdateProfile` semantics from the onboarding slice: the zod schema accepts `nullable().optional()` and the service uses `in` for the filter.

- [2026-07-02] **`summary + chapters` travel as one JSON column write**. The LLM writes both as one `LlmOutputSchema.safeParse` → one Prisma update; user edits must do the same. If a save sends only `summary`, the server rebuilds the full `chaptersJson = {summary, chapters}` from `existing.chaptersJson.chapters ?? []` — otherwise a partial save clobbers the LLM-generated counterpart. The merge logic lives in `apps/api/src/modules/videos/videos.service.ts → updateVideo`.

- [2026-07-02] **Controlled-component tests need a stateful harness**, not a bare mock `onChange`. When the component is `value + onChange` controlled, a mock `onChange={vi.fn()}` doesn't update the rendered value, so test assertions like `findByDisplayValue("New chapter")` after an "Add" click will fail because the new chapter never re-renders. Wrap in a small `<ControlledHarness>` that uses `useState` to mirror real-world behaviour. Pattern used in `apps/web/components/review/chapters-review.test.tsx`.

- [2026-07-02] **Server-rendered detail pages need `router.refresh()` after mutation**, not just TanStack Query invalidation. The page is an RSC that fetches via `serverFetch()`; invalidating the cache doesn't re-run the server fetch. After a successful save in `<VideoMetadataEditor>`, call `router.refresh()` so the header h1 + Details grid reflect the new value. Optimistic local state keeps the editor itself in sync immediately so the next interaction doesn't flash stale text.

- [2026-07-02] **Bug-086 — `updateVideo` partial merge bug**: a save payload containing only `summary` (no `chapters`) must NOT clear the existing chapters. The fix in `apps/api/src/modules/videos/videos.service.ts` is to gate each top-level `data` field on `'key in input` — including the `chaptersJson` write, which has to be re-derived from `existing.chaptersJson ?? {summary: "", chapters: []}` so an absent `chapters` field preserves the prior values while an explicit `null`/`undefined` for `summary` clears it.

- [2026-07-02] **Design.md token discipline pattern.** Every visual property on a dashboard component reaches for `var(--bg|--surface|--ink|--ink-muted|--line|--accent|--status-*)`. When a shadcn primitive (`Badge`, `Button`) doesn't already forward the semantic token from `app/globals.css`, either (a) use `bg-[color:var(--…)]` / `text-[color:var(--…)]` arbitrary values, or (b) drop the primitive and render a plain `<span>` with token classes — do NOT introduce new colors. State (processing/ready/scheduled/error) is signalled by BOTH color AND a text label (Design.md Section 6). Motion is off by default: every animation is gated inside `@media (prefers-reduced-motion: no-preference)` — even Tailwind's `animate-pulse` gets the `motion-safe:` prefix.

- [2026-07-02] **Review page column-width exception.** `docs/Design.md` says text/list content maxes at ~960px (`max-w-[60rem]`). The `/dashboard/published/[id]` review panel is the specified exception — it takes the full width for the two-column video+chapters layout. Details block underneath keeps the 960px cap. - [2026-07-02] **One-shot staggered reveal via CSS custom property.** The `.review-reveal` class in `app/globals.css` reads `--stagger-index` as a per-item inline style (`style={{ ['--stagger-index' as string]: i } as React.CSSProperties}`). - [2026-07-02] **Monospace font for identifiers.** Video IDs, YouTube IDs, timestamps, durations, file sizes, and progress percentages get `font-mono` + `tabular-nums`. - [2026-07-02] **[bug-096] Tailwind v4 plugins must be declared with `@plugin` in CSS, not in `tailwind.config.js`.** `tailwindcss-animate` was in `package.json` but never loaded. In v4 there is no `tailwind.config.js` in this project — every plugin registration goes in `globals.css` as `@plugin "tailwindcss-animate";` immediately after `@import "tailwindcss";`. - [2026-07-02] **Dialog over inline editing for focused mutations.** Chapter editing (title + startMs) belongs in a Dialog — the row is too narrow for safe input, and inline editing blocks the seek-to-chapter affordance. - [2026-07-02] **YouTube status block (privacyStatus / madeForKids / embeddable / license / publicStatsViewable / commentPolicy) was already persisted at create-time but not editable in the review flow.** The `videos.insert` payload builder (`packages/youtube-upload/src/publish-video.ts → buildStatusFromVideo`) reads them off the row, so the missing piece was just (1) extending `updateVideoSchema` to accept the fields, (2) extending the service `updateVideo` `'key in input'` branches, (3) extending `UpdateVideoRequest`, (4) adding UI controls. Inline error in `text-[color:var(--status-error)]` for client-side validation (mirror `validateScheduledAt`: empty OK, `<= now` → "must be in the future", `< 15 min` → YouTube minimum, `> 60 days` → YouTube scheduled max). The user's `defaultTimezone` from settings shows next to the input as a `(Asia/Kolkata)` label, with a hint to change it in Settings → Scheduling. - [2026-07-02] **Test pitfall — fake timers + radix Sheet portals hang `userEvent`.** `vi.useFakeTimers()` in `beforeEach` causes `userEvent.click()` to time out at 5 s when the target is inside a radix `Dialog`/`Sheet` (the portal mounts use `setTimeout` that never fires). Use real timers and wait the redirect delay via `await act(() => new Promise(r => setTimeout(r, 250)))`. Symptom: all tests in a Sheet-touching test file time out at exactly 5000ms with the click never completing. Fix: drop `vi.useFakeTimers()` entirely, use real timers, await any deferred side effects explicitly.

- [2026-07-02] **Test pitfall — `datetime-local` inputs need local-time strings, not UTC strings.** `<input type="datetime-local">` accepts `YYYY-MM-DDTHH:mm` with NO timezone marker, and `new Date("YYYY-MM-DDTHH:mm")` parses as local time per ECMA-262. If the test runner's `process.env.TZ` is non-UTC (e.g. `IST`), `.toISOString().slice(0, 16)` produces a UTC string that, when re-parsed by the validator, gives a different timestamp — sometimes hours in the past. Build the local-time string explicitly: `${y}-${pad(M+1)}-${pad(D)}T${pad(h)}:${pad(m)}` from the Date's local-time getters. Symptom: validation reports "must be in the future" for a Date that was constructed as `Date.now() + 5 * 60 * 1000`.

- [2026-07-02] **Test pitfall — `userEvent.type` on `datetime-local` triggers per-char validation against partial strings.** Each keystroke fires onChange, which triggers the Sheet's `validateScheduledAt`, which can flip the submit button to disabled mid-typing. Use `fireEvent.change(input, { target: { value: futureLocal } })` to set the value atomically in a single React update. Use `userEvent.type` only for inputs that don't have per-keystroke validation (e.g. text inputs that don't gate the submit button).

- [2026-07-09] **Don't wire `prisma migrate deploy` directly into docker-compose when `neon_backup.sql` is in play.** First boot from a dump-loaded Postgres has the final schema but no `_prisma_migrations` table — `migrate deploy` will try to re-CREATE every existing table and crash on `42P07 relation already exists`. Always gate on a schema-existence probe (`SELECT 1 FROM "users" LIMIT 1` here) and baseline via `prisma migrate resolve --applied <dirname>` for each migration in the folder before running `migrate deploy`. The probe target should be a table created by the FIRST migration (and present in the dump), so it's a reliable yes/no for "was this DB bootstrapped from a dump vs. genuinely fresh?". Same gotcha applies anywhere a Postgres container bootstraps from a pg_restore / pg_dump output rather than from the migration history.

## Do-Not-Repeat

- [2026-07-03] Don't put the sidebar in a fixed `h-screen overflow-y-auto` outer wrapper. Mobile browsers fold the chrome bar over the page; `100vh` is taller than the visible viewport and the body becomes unscrollable. Use `min-h-svh` and let the body be the only scroll surface.
- [2026-07-03] Don't ship a "<placeholder> ships with the next slice" disabled button as an empty-state CTA. If the action isn't ready, remove the button entirely and put the explanation in copy. A disabled grey button reads as broken-state to the user.
- [2026-07-04] Don't ship Express 4 routes that depend on async handlers unwrapping their own rejections. Every controller in this codebase returns a Promise; if anything inside `await`s a rejecting call (Prisma, fetch, queue), the response will hang until socket timeout and the only console signal will be `Unhandled promise rejection reason: {}`. The fix is `import "./lib/async-handler.js"` immediately after `import express` in `app.ts` — reorder and the monkey-patch loads after the first `Router()` is built, leaving every controller in the file unprotected. Whenever you add a new top-level `app.use(...)` or `import "express"`, verify the async-handler side-effect import is still the next line.
- [2026-07-04] Don't edit a workspace package's `src/` and assume the consumer will see the change at runtime (bug-138). The `@clipflow/youtube-upload` package exports `./dist/index.js` via `main` + `exports` (see `packages/youtube-upload/package.json`); the API loads the COMPILED output, not the source. After editing `.ts` in any shared package (`packages/youtube-upload`, `packages/s3`, `packages/types`, …) you MUST run `pnpm --filter @clipflow/<pkg> build` before the dev API picks it up — `tsx watch src/index.ts` only reloads the API's own `src/`, not the package graph. Symptom of skipping the rebuild: a known-good `.ts` fix in src paired with a runtime error pointing at `dist/<file>.js` (often `TypeError` because old `dist/` doesn't have the unwrap/normalization you added). Whenever your ChangeSet touches a package whose `main` points at `dist/`, rebuild that package before testing the consumer.

- [2026-07-04] Don't cast Prisma's JSON columns to a typed shape at the read site without first verifying the writer's actual contract (bug-125). The `chaptersJson` column was assumed to be `Chapter[]` in publish-video.ts but was actually `{ summary, chapters[] }` from the LLM path. The TS cast was structurally valid (the wrapper object can be cast to anything), so the type-check passed. At runtime, `formatChaptersForDescription` received an object, walked `.summary` + `.chapters` as if they were chapter rows, and emitted `NaN:NaN` lines. No error surfaced — YouTube just silently dropped the malformed timestamps. Whenever you read a Prisma `Json?` column, write a one-line runtime shape check (or use a zod parser like `videos.types.ts → parseChaptersJson`) instead of trusting a structural cast. Bug class: "looks type-safe, fails silently".

- [2026-07-04] Express 4 does not forward async handler rejections to `next(err)`. Every controller in this codebase is `async (req, res) => { ... }` with no try/catch — when a Prisma call rejects, the response never goes out and the request hangs until socket timeout. The only console signal is `Unhandled promise rejection reason: {}` (pino collapses Prisma errors because they expose `message`/`stack` via getters the default serializer can't reach). Express 5 fixed this natively; until we migrate, the patch lives in `apps/api/src/lib/async-handler.ts` and monkey-patches `Layer.prototype.handle_request` to `.catch(next)` any returned promise. Import order in `app.ts` is load-bearing: `import express` first, `import "./lib/async-handler.js"` next (side-effect import, no binding needed), then any router imports. Reorder and you get an unpatched Layer prototype at `Router()` construction time.
- [2026-07-04] The pino default serializer collapses Prisma errors to `{}` because their `.message` / `.stack` properties are getters on the prototype chain, not enumerable own properties. Don't trust `{ reason }` in an unhandledRejection log — pipe `reason` through `util.inspect(reason, { depth: 6, breakLength: Infinity })` first (and if it's an Error, also surface `.stack` separately so the trace survives log-rotation compaction). This was the difference between "POST hangs, server says nothing useful" and "POST hangs, server prints the Prisma error with column names + line numbers".
- [2026-07-07] **Don't assume there's a Prisma relation for every `@unique` FK column.** `Video.selectedThumbnailId` is a unique FK to `Thumbnail.id`, but the schema deliberately leaves the join to the service layer (`packages/db/schema.prisma:472-474`: "Application-resolved (no Prisma relation needed — the service layer handles consistency)"). Trying to `include: { selectedThumbnail: true }` in a query throws at runtime. Use `prisma.thumbnail.findUnique({ where: { id: video.selectedThumbnailId } })` instead — and remember to validate `thumbnail.videoId === video.id` since the column is unique but the lookup is key-based.
- [2026-07-07] **Don't reach for `vi.doMock` after the SUT has already imported the module.** Vitest hoists `vi.mock(..., { factory })` calls to the top of the file, but `vi.doMock` is runtime-only and can't replace a module already imported via static `import`. To stub a hook the SUT uses (like `useApi`), prefer `vi.mock("@/hooks/use-api", () => ({ useApi: () => stub }))` at the top of the test file — every render inside the file gets the stub without `await import()` gymnastics. Symptom of getting this wrong: your stubs never seem to fire, the real hook runs, and the SUT makes network calls you can't intercept.
