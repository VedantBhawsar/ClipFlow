## Decision Log

<!-- Significant technical decisions with rationale. Why X was chosen over Y. -->

- [2026-06-26] /dashboard/published was rewritten to mirror the dashboard's row style (thumbnail + title + meta line + "View on YouTube" button) rather than a custom 4-column editorial grid. Rationale: design.md says single-column, max-width ~960px for text/list content. The StatusTimeline strip is the signature element on the in-flight dashboard, but the published library lists *finished* work — a filled timeline is noise here, so we drop it and rely on the privacy pill + published date.

- [2026-06-26] Don't render the StatusTimeline strip on views where every video is at the final stage (e.g. /dashboard/published, where status === "PUBLISHED" by definition). Show pipeline progress only where progress is actually happening.

- [2026-06-26] Custom-thumbnail upload is wired end-to-end: user picks a JPEG/PNG (≤2 MB) in the create-video dialog → `POST /api/videos` mints a second presigned POST URL → browser uploads in parallel with the video bytes → `finalizeUpload` HEADs both objects and persists `s3KeyThumbnail` + `thumbnailContentType` on the row → after the worker calls `videos.insert` successfully, it streams the S3 object to `POST /upload/youtube/v3/thumbnails/set`. The thumbnail block in `CreateVideoResponse` is `null` when the user didn't pick one — the rest of the pipeline is unchanged for that path.

- [2026-06-26] The `youtube.service.test.ts` failures (`TypeError: res.text is not a function`) are pre-existing — the mock chain uses `res.json` where the production code calls `res.text`. Unrelated to anything in the videos module. Don't touch it as part of a videos-task fix.

- [2026-06-26] Zod `superRefine` functions must avoid indexing the typed input by a string key — TS7053. Use explicit `if (data.x !== undefined) …` per field instead of `data[k]` or `present.includes(k)`. Future zod refinements in this codebase: no string-keyed access.

- [2026-06-26] Custom thumbnail is OPTIONAL on the Video row (`s3KeyThumbnail`, `thumbnailContentType` both nullable). When the user skips it, the publish path falls back to YouTube's generated frame. Don't gate the publish flow on having a thumbnail — a partial upload of just a video must still succeed.

- [2026-06-26] YouTube `thumbnails.set` accepts ONLY JPEG/PNG up to 2 MB. Both bounds are enforced at the create edge (`videos.schemas.ts`), the runtime (`videos.service.ts` — caps the presigned POST to `YOUTUBE_MAX_THUMBNAIL_BYTES`), and the client form (zod `.refine()`). Mirror any new image-upload bound in all three layers.

- [2026-06-26] Thumbnail upload failures (transient OR permanent) are logged but do NOT fail the publish — the video is live with YouTube's default frame, which is a valid outcome. Only re-throw transient errors so BullMQ retries; permanent errors get swallowed after logging.

- [2026-06-27] Worker startup-recovery runs in two passes. First pass reconciles rows orphaned in `PUBLISHING` by a crashed worker: if `youtubeVideoId` is set, finalize to `PUBLISHED` (upload likely completed on YouTube's side, only the DB flip was lost); if not, reset to `READY` so the standard recovery re-enqueues it. Only AFTER orphans are reconciled does the `READY`/`SCHEDULED` re-enqueue pass run. Rationale: a `PUBLISHING` row that gets reset to `READY` becomes eligible for the second pass in the same boot, so a crashed-mid-upload job is recovered without waiting for the next failure tick.

- [2026-06-27] When wiring a new filter / search / sort param through the web stack, ALL four layers need an explicit change: (1) server schema (`zod` query), (2) service-layer Prisma `where`, (3) shared type (`ListXParams`), (4) web `api-client.ts` query-string builder. A working (1)+(2)+(3) with a missing `search.set(...)` in (4) silently passes the empty filter to the server — the user sees the unfiltered list and assumes the filter is broken. Always grep for `if (params?.X) search.set` in `api-client.ts` when adding a filter param.

- [2026-06-28] Cache abstraction uses two backends (Redis when REDIS_URL is set, in-memory fallback). The exported `cache` singleton is a delegating object — `get/set/del` forward to whichever backend `initCache(env)` selected — so every service call site (`cache.get("settings:${userId}")` in settings/youtube/videos) keeps working unchanged. Tests that `vi.mock("../../lib/cache.js", () => ({ cache: { get, set, del } }))` still work because the mock replaces the whole module surface. New lifecycle: `initCache(env)` (boot, idempotent) → `verifyCache(env)` (PINGs Redis, returns latency) → `disposeCache()` (async; calls `redis.quit()` for Redis backend, clears the in-memory sweeper for the fallback). Boot-time service banner in both `apps/api/src/index.ts` and `apps/worker/src/index.ts` runs Database (`SELECT 1`) + Cache (Redis PING) + Queue (BullMQ PING) probes BEFORE `listen()`, prints ✓/✗ with latency, and exits non-zero if a required service is unreachable. Optional services (in-memory cache, queue when REDIS_URL unset) show as "skipped". Worker side opens a fresh `ioredis` probe connection (separate from the BullMQ one) so the PING is independent of queue lifecycle.

- [2026-06-28] ioredis needs `lazyConnect: true` when you want to control connection timing at boot. Combined with `maxRetriesPerRequest: null` (required for BullMQ compatibility), this lets you call `redis.connect()` + `redis.ping()` once at startup, fail fast if Redis is unreachable, and never accidentally issue a command before the socket is up. The `error` event handler must be a no-op (or log) — ioredis will emit errors mid-reconnect and you don't want them to crash the process.

- [2026-06-30] **pnpm + Docker pitfall.** pnpm uses isolated mode by default — each workspace package's direct deps live under `apps/<pkg>/node_modules/<dep>` as *relative symlinks* into the virtual store at `node_modules/.pnpm/`. A Dockerfile that copies only the root `/app/node_modules` (without `apps/<pkg>/node_modules`) will let the runtime process find `.pnpm/<dep>` but NOT `<dep>` itself — Node walks up looking for `node_modules/<dep>` from the entrypoint and finds nothing. Symptom: `ERR_MODULE_NOT_FOUND: Cannot find package '<dep>'`. Fix: every Dockerfile runtime stage that runs a pnpm-managed workspace package must `COPY --from=build apps/<pkg>/node_modules ./apps/<pkg>/node_modules` alongside the root `node_modules`. Either that, or `pnpm deploy` (single self-contained output dir), or `.npmrc` `public-hoist-pattern[]=<dep>` to hoist specific runtime deps to the root. The bug is latent until the package gains a direct import of one of its deps — BullMQ was previously imported indirectly (BullMQ's own nested node_modules carried `ioredis`), so it shipped fine without the apps/worker/node_modules layer; once apps/worker/src/index.ts started doing `import { Redis } from "ioredis"` directly, the missing COPY became fatal.

- [2026-06-30] **Worker BullMQ broker split (silent).** API on host + worker in compose must share the same BullMQ broker. `apps/api/.env` and `apps/worker/.env.docker` both point at Redis Cloud (`db.redis.io:10367`); the compose worker's `environment:` block has an explicit `REDIS_URL:` override as a belt-and-braces. Do NOT change that override to `redis://redis:6379` (the local compose Redis). Symptom of the split: video row sits in `EXTRACTING` forever, worker logs show zero job activity (not even a job pull attempt), `clipflow:video-ingest:wait` is empty on the worker's broker but populated on the API's. Diagnostic: `docker compose config | grep -A 12 worker:` and compare `REDIS_URL` to `apps/api/.env`. If they don't match, jobs are stranded on the wrong broker — there is no log signal of the split on either side.

- [2026-07-01] Auth.js v5 middleware default-export typing: the inferred return type of `NextAuth(config).auth` references next-auth's private `lib/` and `lib/types` subpaths, which TS2742 cannot name portably. Always export as `NextAuth(authConfig).auth as NextMiddleware` (importing `NextMiddleware` from `next/server`). The cast is purely structural — runtime contract is identical, but TypeScript stays happy. Documented alongside the do-not-repeat for `headers()` out of scope.

- [2026-07-01] `apps/web/middleware.ts` must export `NextAuth(authConfig).auth` as a function reference, NOT invoked (no `()`). Next.js 16 + Turbopack bundles middleware.ts with the RSC path, so calling `.auth()` eagerly dispatches into Auth.js v5 code that calls `next/headers`'s `headers()` — which only works inside a request scope. Eager invocation crashes at module load with `` `headers` was called outside a request scope ``. Fix: `export default NextAuth(authConfig).auth as NextMiddleware;`. The `as NextMiddleware` cast is required because the inferred type of `.auth` can't be named without referencing next-auth's private `lib/` and `lib/types` subpaths (TS2742).

- [2026-07-01] Adding nullable columns to a Prisma model that has a typed test-fixture alias breaks `tsc --noEmit` at the call sites. After every `schema.prisma` change that adds a column to `Video` (or any model with a hand-rolled `StubX` / `mockX` test alias), grep for the alias name and add matching nullable fields to BOTH the type alias and every inline row literal in `apps/api/src/modules/videos/videos.service.test.ts` + `apps/api/src/modules/auth/auth.service.test.ts` (the latter breaks whenever the `Env` type gains a new field). For `Json?` columns the field type must be `Prisma.JsonValue | null` — `unknown` fails assignability. Bug-029 / bug-055 (recurring). The repo has no machinery for this yet; if it becomes a third occurrence, write a vitest fixture generator that reads `prisma generate`'s output and emits the typed stubs.

- [2026-07-01] **vitest `vi.mock` factory TDZ bug** (bug-071). `vi.mock("@module", () => ({ fn: mockFn }))` is hoisted above module-level `const mockFn = vi.fn()` declarations by vitest's transformer. When the test RUNS, the factory closure hits the TDZ on `mockFn` and the whole test file fails with `ReferenceError: Cannot access 'mockFn' before initialization`. The fix is `vi.hoisted(() => ({ mockFn: vi.fn(), ... }))` for all mock declarations — `vi.hoisted` itself is hoisted above `vi.mock` factories. Note: this only surfaces when `it()` actually executes; tests with `it.skip` (like `video-ingest.test.ts`) hide the problem. When adding new test files with mocks, always start with `vi.hoisted` instead of `const` so the lint pass / future un-skipping doesn't suddenly break.

- [2026-07-01] **Don't write new lib sub-features to `apps/<feature>/`** — every lib lives INSIDE an app's `src/lib/` (bug-080). The convention is `apps/<pkg>/src/lib/<feature>/`. The `apps/llm/` directory I created was outside any app's `src/` — orphaned code with no package.json to scope it. Always run `ls apps/<pkg>/src/lib/` before Write on a new feature directory.

- [2026-07-01] **OpenAI SDK v6+ error class Headers arg is non-optional** (bug-083). `new AuthenticationError(status, error, message, headers)` — the 4th arg is `Headers`, not `undefined`. Every test fixture must construct a `new Headers()` (empty is fine) and pass it. The SDK treats `undefined` as a type error, not a permissive `null`. Symptom: every error subclass construction in tests fails TS2345.

- [2026-07-01] **Prisma `InputJsonValue` requires an explicit index signature** (bug-084). A typed interface like `interface ChaptersJson { summary: string; chapters: ...[] }` is NOT assignable to `InputJsonValue` — Prisma's type wants `[key: string]: unknown`. The pattern that works: keep the typed interface in the function body, cast to `unknown as object` ONLY at the `prisma.update({ data: { jsonColumn: x as unknown as object } })` call site. The runtime serialization is identical; the cast is purely nominal.

- [2026-07-01] **NVIDIA NIM via the OpenAI SDK, not a separate adapter**. NVIDIA's NIM endpoint (`integrate.api.nvidia.com/v1`) is wire-compatible with OpenAI's chat completions shape — same request/response, same `response_format: { type: "json_object" }` flag, same streaming semantics. Decision: use the `openai` npm package and override `baseURL` to NVIDIA's endpoint, instead of pulling in a separate NVIDIA SDK or hand-rolling HTTP. Benefits: one client class (`OpenAICompatLlmClient`), one set of error classifications (`classifyLlmError` works against both), one retry helper. The wrapper branches on `LLM_PROVIDER === "nvidia"` vs `"openai"` only at construction (NVIDIA needs `baseURL`, OpenAI doesn't). Future Anthropic support will need a sibling wrapper — different SDK, different wire format.

- [2026-07-01] **Worker-side enqueue vs API-side enqueue for follow-up jobs**. The v1.5 ingest → transcription pipeline enqueues `transcribe-${videoId}` from inside the `video-ingest` BullMQ worker (after the row is flipped to `TRANSCRIBING`), NOT from the API at video-create time. Reason: with concurrency=1 on each queue, a worker's own follow-up fires only AFTER the parent completes, so the deterministic `s3KeyAudio` guarantee holds without any polling. If the API chained them, the transcription job would either (a) need to poll for `s3KeyAudio` (ugly), or (b) race against the still-running ingest and fail permanent with `[AAI_AUDIO_MISSING]` (worse — it would cause false-positive permanent failures). The worker-side `enqueueTranscription` callback is passed through `ProcessContext` and is optional so existing tests can pass a no-op. API-side `enqueueTranscriptionJob` still exists in `apps/api/src/lib/queue.ts` for admin-style retry paths (e.g. retry-from-FAILED).

- [2026-07-01] **Dashboard needs an SSE fallback, not just SSE.** The dashboard's `useVideoSSE` is the real-time path, but it's a single point of failure: if Redis blips, EventSource silently disconnects without an obvious UI signal, or the worker/API channel-prefix drifts, the row stays stuck on its old badge forever (bug-081). Solution: pair SSE with a `setInterval` safety-net refetch, gated on `hasInFlightVideo` (any non-final status). 15 s feels live without hammering the server. Interval clears the moment no in-flight rows remain — free in the steady state. Always `useMemo` the array dependency so the polling effect doesn't tear down + recreate every render.

- [2026-07-01] **Single LLM call for {summary, chapters[]} beats two separate calls.** Putting summary + chapter selection in one prompt cuts latency in half, costs roughly the same as one larger prompt (vs. two), and keeps the chapter boundaries + summary internally consistent — they reason over the same context in the same forward pass. The output schema `LlmOutputSchema` enforces YouTube rules (first chapter = 0, ≥10 s gap, ≤12 chapters, ≤100 char titles, ≤280 char summary) server-side via zod `.refine()` so the LLM gets a focused system prompt and we get the safety net.

- [2026-07-01] **LLM retries belong in the caller, not the client.** The OpenAI SDK wrapper (`OpenAICompatLlmClient.complete`) is a thin facade: it makes one call, unwraps the text, surfaces the SDK error class. It does NOT classify or retry. Classification (`classifyLlmError` → `{ kind, reasonCode, message }`) and parse-retry (`validateWithRetry`, max 3 attempts with the rejection reason appended to the next userPrompt) both live in the `generate` job's call site. The thin client is trivially mockable; the caller owns the "permanent vs transient" decision.

- [2026-07-01] **`chaptersJson` schema changed shape mid-flight** (no migration needed). Originally declared as a bare chapter array. v1.5 adds a `summary` field alongside chapters — so it becomes `{ summary, chapters[] }`. The publish path reads `chaptersJson.chapters` (still works), the LLM path writes both fields (one `LlmOutputSchema.safeParse` succeeds, single JSON column write). Why no migration: Prisma's `Json?` column accepts any JSON shape; only the schema comment + TypeScript type changed. If we ever want to validate the JSON shape at the DB layer, switch to Postgres's typed JSON or zod on read at every consumption point.

- [2026-07-01] **vitest `mock.calls` destructure off-by-one** (bug-085). `mock.calls[0]` for a single-arg call is `[{...}]` — one inner array containing one element. `[, args]` skips that element and captures index 1 (always `undefined`). The bug hides in `.filter()` calls (predicate is false for `undefined.args.data`, filter returns `[]`, `toHaveLength(0)` passes by luck). It surfaces in `.map()` calls. Single-arg mock calls should always use `([args])`, not `([, args])`. Audit pattern when reviewing test files.

- [2026-07-02] **Partial-update endpoints must check `'key' in input`**, not `input.key !== undefined`, so explicit `null` survives the merge (a creator clearing the description to `null` shouldn't accidentally leave the old text in place). Mirror the `useUpdateProfile` semantics from the onboarding slice: the zod schema accepts `nullable().optional()` and the service uses `in` for the filter.

- [2026-07-02] **`summary + chapters` travel as one JSON column write**. The LLM writes both as one `LlmOutputSchema.safeParse` → one Prisma update; user edits must do the same. If a save sends only `summary`, the server rebuilds the full `chaptersJson = {summary, chapters}` from `existing.chaptersJson.chapters ?? []` — otherwise a partial save clobbers the LLM-generated counterpart. The merge logic lives in `apps/api/src/modules/videos/videos.service.ts → updateVideo`.

- [2026-07-02] **Controlled-component tests need a stateful harness**, not a bare mock `onChange`. When the component is `value + onChange` controlled, a mock `onChange={vi.fn()}` doesn't update the rendered value, so test assertions like `findByDisplayValue("New chapter")` after an "Add" click will fail because the new chapter never re-renders. Wrap in a small `<ControlledHarness>` that uses `useState` to mirror real-world behaviour. Pattern used in `apps/web/components/review/chapters-review.test.tsx`.

- [2026-07-02] **Server-rendered detail pages need `router.refresh()` after mutation**, not just TanStack Query invalidation. The page is an RSC that fetches via `serverFetch()`; invalidating the cache doesn't re-run the server fetch. After a successful save in `<VideoMetadataEditor>`, call `router.refresh()` so the header h1 + Details grid reflect the new value. Optimistic local state keeps the editor itself in sync immediately so the next interaction doesn't flash stale text.

- [2026-07-02] **Bug-086 — `updateVideo` partial merge bug**: a save payload containing only `summary` (no `chapters`) must NOT clear the existing chapters. The fix in `apps/api/src/modules/videos/videos.service.ts` is to gate each top-level `data` field on `'key in input` — including the `chaptersJson` write, which has to be re-derived from `existing.chaptersJson ?? {summary: "", chapters: []}` so an absent `chapters` field preserves the prior values while an explicit `null`/`undefined` for `summary` clears it.
